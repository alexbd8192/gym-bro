# Gym Bro — Dev Guide

## How it works

The app has two pieces that must both be running during development:

| Piece | What it is | Port |
|---|---|---|
| **Vite** | Serves the React frontend | 5173 |
| **PocketBase** | Backend database + auth API | 8090 |

Vite automatically proxies any `/api/` or `/_/` request to PocketBase, so the app only ever talks to one URL (`localhost:5173`).

---

## Starting the dev environment (local — no Docker)

Open **two terminals** in the project folder:

**Terminal 1 — PocketBase:**
```bash
./pocketbase serve
```

**Terminal 2 — Vite:**
```bash
npm run dev
```

Then open **http://localhost:5173**.

> `docker-compose.yml` is for the production server only.
> Local dev runs Vite + PocketBase directly — faster and no rebuild needed on every change.

> The `pocketbase` binary is not in git — download it once from https://pocketbase.io/docs/
> and drop it in the project root. It's a single file, no install needed.

---

## Access from another device (e.g. your partner's phone)

While both servers are running, connect to the same Wi-Fi and open:
**http://192.168.37.250:5173**

---

## User accounts

Accounts are managed in the PocketBase admin panel — **not** in the app itself.

1. Go to **http://localhost:8090/_/**
2. Collections → **users** → New record
3. Fill in email + password

The first time you open `/_/` you'll be asked to create a superuser (admin) account — that's just for the admin panel, not the app.

## Project structure

```
gym-bro/
│
├── gymbro.tsx              ← entire app UI + logic (React, one big file)
│
├── src/
│   ├── main.tsx            ← entry point, mounts React (don't touch)
│   ├── pb.ts               ← PocketBase client singleton (don't touch)
│   └── api.ts              ← all database operations (CRUD functions)
│
├── pb_migrations/          ← database schema & rules, applied automatically
│   │                          on PocketBase startup (newest = highest number)
│   ├── 1744934400_init.js              ← creates all 4 collections
│   ├── 1744934401_api_rules.js         ← access rules (users see only their data)
│   ├── 1744934402_profile_extended_fields.js  ← adds weight/measurement history fields
│   └── 1776509*_updated_*.js           ← auto-generated by PocketBase admin UI
│
├── Dockerfile              ← builds the frontend for production
├── nginx.conf              ← serves the app + proxies PocketBase in production
├── docker-compose.yml      ← runs frontend + PocketBase together (production)
│
├── vite.config.ts          ← Vite config, dev proxy rules (don't touch)
├── index.html              ← HTML shell (don't touch)
├── package.json            ← dependencies
└── DEV.md                  ← this file
```

### The 4 database collections

| Collection | What's stored |
|---|---|
| **profiles** | One record per user — measurements, plate inventory, theme, settings |
| **programs** | Training blocks (PPL, 5/3/1, etc.) that group routines |
| **routines** | Named workout templates with exercises, sets, reps |
| **sessions** | Every logged workout (strength, cardio, or rest day) |

All records have a `userId` field — users can only read/write their own data.

## Production deploy (homelab)

```bash
docker compose up -d
```

Runs on port 80. PocketBase is not exposed to the host — only the nginx container can reach it.
